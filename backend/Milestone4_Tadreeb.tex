\section{Backend API Development Overview}

The \textbf{Tadreeb} backend is a RESTful API built with \textbf{Node.js} and \textbf{Express.js}, backed by \textbf{MongoDB} via the Mongoose ODM. It serves as the data and business-logic layer for the Tadreeb platform, which connects university students with companies offering internship and co-op opportunities.

\subsection*{Technology Stack}
\begin{itemize}[leftmargin=1.5cm]
  \item \textbf{Runtime:} Node.js (v18+)
  \item \textbf{Framework:} Express.js
  \item \textbf{Database:} MongoDB (Mongoose ODM)
  \item \textbf{Authentication:} JSON Web Tokens (JWT) — HS256, 7-day expiry
  \item \textbf{Password hashing:} bcrypt (salt rounds: 10)
  \item \textbf{CORS:} configurable via \texttt{CORS\_ORIGIN} environment variable
\end{itemize}

\subsection*{Base URL}
\begin{lstlisting}
http://localhost:5001/api
\end{lstlisting}

\subsection*{Route Groups}
\begin{itemize}[leftmargin=1.5cm]
  \item \texttt{/api/auth} — student/company/admin authentication
  \item \texttt{/api/opportunities} — internship program CRUD
  \item \texttt{/api/applications} — student applications lifecycle
  \item \texttt{/api/admin} — admin management operations
  \item \texttt{/api/company-requests} — public company join requests
\end{itemize}

\subsection*{Project Structure}
\begin{lstlisting}

\section{Authentication}

\subsection*{Overview}

The API uses \textbf{JWT (JSON Web Token)} authentication. On a successful login or registration, the server returns a signed token the client must store and include in subsequent protected requests.

\begin{itemize}[leftmargin=1.5cm]
  \item \textbf{Algorithm:} HS256
  \item \textbf{Expiry:} 7 days
  \item \textbf{Secret:} read from \texttt{process.env.JWT\_SECRET} (server exits on startup if missing)
\end{itemize}

\subsection*{Sending the Token}

All protected routes require the token in the \texttt{Authorization} header:

\begin{lstlisting}
Authorization: Bearer <token>
\end{lstlisting}

\subsection*{Token Generation (\texttt{utils/generateToken.js})}

\begin{lstlisting}[language=JavaScript]
const jwt = require('jsonwebtoken');

const generateToken = (id, role) =>
  jwt.sign({ id, role }, process.env.JWT_SECRET, { expiresIn: '7d' });

module.exports = generateToken;
\end{lstlisting}

\subsection*{Auth Middleware (\texttt{middleware/authMiddleware.js})}

\begin{lstlisting}[language=JavaScript]
const jwt = require('jsonwebtoken');

const protect = (req, res, next) => {
  const authHeader = req.headers.authorization;
  if (!authHeader || !authHeader.startsWith('Bearer '))
    return res.status(401).json({ message: 'Not authorized, no token' });

  try {
    const token = authHeader.split(' ')[1];
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded; // { id, role }
    next();
  } catch {
    res.status(401).json({ message: 'Not authorized, token failed' });
  }
};

module.exports = { protect };
\end{lstlisting}

\subsection*{Role Middleware (\texttt{middleware/roleMiddleware.js})}

\begin{lstlisting}[language=JavaScript]
const allowRoles = (...roles) => (req, res, next) => {
  if (!roles.includes(req.user.role))
    return res.status(403).json({ message: 'Access denied' });
  next();
};

module.exports = { allowRoles };
\end{lstlisting}

\begin{center}
\begin{tabular}{ll}
  \toprule
  \textbf{Role} & \textbf{Capabilities} \\
  \midrule
  \texttt{student} & Register, login, view opportunities, apply, cancel application, view own applications \\
  \texttt{company} & Login, create/update/delete own opportunities, view \& manage applications \\
  \texttt{admin}   & Create companies/students/admins, view all users, update company status \\
  \bottomrule

\end{center}

\subsection*{Password Hashing}

Passwords are hashed with \textbf{bcrypt} (salt rounds: 10) before being stored. Plain-text passwords are never persisted.

\begin{lstlisting}[language=JavaScript]
const bcrypt = require('bcryptjs');
const hashed = await bcrypt.hash(password, 10);
// verification
const match = await bcrypt.compare(plaintext, hashed);
\end{lstlisting}

\subsection*{Input Validation (\texttt{utils/validators.js})}

\begin{lstlisting}[language=JavaScript]
const EMAIL_REGEX = /^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$/;

const isValidEmail    = (email)    => typeof email === 'string' && EMAIL_REGEX.test(email);
const isStrongPassword = (pwd)     => pwd.length >= 8 &&
                                      /[A-Za-z]/.test(pwd) && /[0-9]/.test(pwd);
const isValidObjectId  = (id)      => mongoose.Types.ObjectId.isValid(id) &&
                                      String(new ObjectId(id)) === id;
const exceedsMaxLength = (str, n)  => typeof str === 'string' && str.length > n;
\end{lstlisting}

\subsection*{Server-Level Guards (\texttt{server.js})}

\begin{itemize}[leftmargin=1.5cm]
  \item \textbf{Content-Type enforcement:} \texttt{POST}/\texttt{PUT}/\texttt{PATCH} requests with a non-JSON body receive \texttt{415 Unsupported Media Type}.
  \item \textbf{404 catch-all:} any unrecognised \texttt{/api} route returns \texttt{404 Route not found}.
  \item \textbf{Central error handler:} malformed JSON → 400, Mongoose \texttt{CastError} → 400, \texttt{ValidationError} → 400, duplicate key (code 11000) → 400.
  \item \textbf{JWT\_SECRET guard:} server calls \texttt{process.exit(1)} on startup if the secret is absent.
\end{itemize}

% ── APPLICATIONS ──────────────────────────────────────────────
\subsection*{3.3 Application Routes — \texttt{/api/applications}}

\subsubsection*{POST /api/applications \quad 🔒 Student only}

Submits a new application. Business rules enforced:
\begin{itemize}[leftmargin=1.5cm]
  \item Program must exist and have \texttt{status = "Active"}
  \item Student cannot apply twice to the same program (DB unique index + controller check)
  \item \textbf{Seat capacity:} if accepted count $\geq$ program seats → 400 ``Program is full''
  \item \textbf{Overlap guard:} if the student already has an \texttt{Accepted} application whose date range overlaps the new program's range → 400 ``You already have an accepted program during this period''
\end{itemize}

\textbf{Request Body:}
\begin{lstlisting}[language={}]
{ "programID": "<opportunity_id>" }
\end{lstlisting}

\textbf{Success 201:}
\begin{lstlisting}[language={}]
{
  "message":     "Application submitted successfully",
  "application": {
    "_id":          "...",
    "studentID":    "...",
    "programID":    "...",
    "status":       "Submitted",
    "appliedDate":  "2025-04-25T00:00:00.000Z",
    "decisionNote": "-"
  }
}
\end{lstlisting}

\subsubsection*{PATCH /api/applications/:id/status \quad 🔒 Company only (must own program)}

Updates an application's status. Business rules enforced:
\begin{itemize}[leftmargin=1.5cm]
  \item Ownership: the program the application references must belong to the company making the request
  \item \textbf{Seat re-check before accepting:} if seats are full a second \texttt{Accepted} is blocked
  \item \textbf{Auto program-status sync:} when accepting causes accepted count $=$ seats → \texttt{program.status = "Completed"}; when rejecting causes accepted count $<$ seats → \texttt{program.status = "Active"}
\end{itemize}

\textbf{Request Body:}

\end{lstlisting}

\textbf{status} must be one of: \texttt{"Submitted"}, \texttt{"Under Review"}, \texttt{"Accepted"}, \texttt{"Rejected"}\\
\textbf{decisionNote} is optional (saved even as empty string).

\begin{lstlisting}[language={}]

\end{lstlisting}

\begin{lstlisting}[language={}]
